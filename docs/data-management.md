# 학습 데이터 관리 시스템 문서 (Data Management System)

이 문서는 MyVoca 서비스에서 사용자의 학습 데이터가 어떻게 초기화되고, 변경되며, 각 서비스(Card, Quiz, Voca)에서 어떻게 활용되는지 설명합니다.

---

## 1. 데이터 초기화 (Initialization)

사용자가 처음 서비스를 시작하거나 온보딩 과정을 거칠 때 `initAppData` 함수를 통해 데이터가 생성됩니다.

- **절차**:
    1. **단어 목록 로드**: Supabase `Word` 테이블에서 전체 단어 ID를 가져옵니다.
    2. **셔플링 및 그룹화**: 단어 ID를 무작위로 섞은 후, 설정된 번들 크기(기본 30개)로 나누어 `Day` 단위의 `wordMap`을 생성합니다.
    3. **사용자 데이터 생성**: 시작 시간, 연속 학습일, 오늘 학습량 등을 포함하는 `userData` 객체를 생성합니다.
    4. **저장 및 동기화**:
        - 모든 데이터는 우선 로컬 스토리지(`localStorage`)에 저장됩니다.
        - 만약 사용자가 OAuth(카카오 등)로 로그인된 상태라면, 생성된 `wordMap`의 모든 단어 상태를 Supabase `Voca` 테이블에 `upsert`하여 동기화합니다.

---

## 2. 데이터 변경 절차 (Data Modification)

사용자의 학습 진행 상황은 `updateWordStatus` 함수를 통해 실시간으로 업데이트됩니다.

- **변경 흐름**:
    1. **로그인 상태 확인**: `supabase.auth.getSession()`을 통해 현재 로그인 여부를 판단합니다.
    2. **DB 업데이트 (OAuth 유저)**: 
        - Supabase `Voca` 테이블에서 해당 유저의 특정 단어(`word_id`)의 `status`를 `true`로 업데이트합니다.
    3. **로컬 업데이트 (Guest 유저)**:
        - 로컬 스토리지의 `wordMap`에서 해당 단어가 포함된 `Day`를 찾습니다.
        - 해당 Day의 `finishedCount`를 증가시키고, 전체 길이에 따른 `progress` 및 `done` 상태를 재계산하여 저장합니다.

---

## 3. 서비스별 데이터 활용 (Service Usage)

### 3.1 암기하기 (Play/Card)
- **데이터 로드**: `useWord` 훅을 통해 현재 선택된 Day의 단어 리스트를 가져옵니다.
- **상태 반영**: 사용자가 카드를 넘길 때마다(`nextCard`), 해당 단어의 학습 상태를 `true`로 업데이트하여 '학습 완료'로 처리합니다.

### 3.2 퀴즈풀기 (Play/Quiz)
- **데이터 로드**: 현재 Day의 단어 중 **학습이 완료되지 않은(done: false)** 단어들만 필터링하여 퀴즈 목록(`quizs`)을 구성합니다.
- **상태 반영**: 사용자가 퀴즈의 정답을 맞혔을 때(`handleCorrect`), 해당 단어의 학습 상태를 `true`로 업데이트합니다.

### 3.3 단어장 (Voca)
- **데이터 로드**: `loadUserData` 로더에서 가공된 `wordMap` 데이터를 사용합니다.
- **진행도 표시**: 
    - 각 Day별로 `(완료된 단어 수 / 전체 단어 수)` 비율을 계산하여 `progress` 바를 표시합니다.
    - 모든 단어가 완료되면 해당 Day는 '완료' 상태로 시각화됩니다.

---

## 4. 데이터 마이그레이션 (Migration)

게스트 모드로 사용하던 사용자가 로그인을 할 경우, `migrateLocalDataToSupabase` 함수가 실행되어 로컬에 저장되어 있던 학습 기록을 Supabase DB로 통합합니다. 이를 통해 기기 변경 시에도 학습 데이터를 유지할 수 있습니다.

---

## 5. 알림(Notification) 시스템

알림 시스템은 시스템 공지와 사용자 개인 알림을 관리합니다.

- **데이터 스키마**: `public.Notification` 테이블에 `title`, `content`, `type`, `user_id` 등의 구조로 알림 데이터를 저장합니다.
- **데이터 흐름**:
    1. 클라이언트의 `NotificationList` 컴포넌트가 마운트될 때 Supabase에서 알림 데이터를 최신순(`created_at` 내림차순)으로 Fetch합니다.
    2. RLS 정책을 통해 전체 사용자 대상 공지(`user_id IS NULL`)와 현재 로그인한 사용자 본인의 알림만 수신합니다.
    3. 클라이언트 코드(Front-end) 레벨에서 로그인 여부에 따른 필수 시스템 알림(예: 데이터 동기화 권장, 동기화 완료 상태)을 Fetch된 데이터 상단에 병합(Merge)하여 노출합니다.
